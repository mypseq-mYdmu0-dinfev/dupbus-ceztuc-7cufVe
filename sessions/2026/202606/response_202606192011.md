# Response to "concisely tell how to use `/schedule` to send a msg at an exact time (SYD timezone) to existing session (not new)"

1. **Short answer: not directly possible.** `send_message` (the tool that targets an existing session) is blocked in unsupervised/auto mode — which is exactly how scheduled tasks run. So a `/schedule`-fired task cannot deliver to an existing session.

2. What `/schedule` CAN do at an exact time:
- 2.1. Use `fireAt` with an ISO 8601 timestamp + SYD offset
- 2.2. June–Sept (AEST, UTC+10): `"2026-06-20T09:00:00+10:00"`
- 2.3. Oct–Apr (AEDT, UTC+11): `"2026-11-01T09:00:00+11:00"`
- 2.4. The task fires once, auto-disables, and spawns a **new** session

3. Closest workaround if you need it in an existing session:
- 3.1. Schedule the task to write output to a known file (e.g. `sessions/2026/202606/sched_output.md`)
- 3.2. Resume the target session after the file appears
