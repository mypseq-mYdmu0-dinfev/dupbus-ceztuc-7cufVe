5. Re 2.1 (selecting the interpreter) —— you don't need to do it manually
- 5.1. The `GH.code-workspace` already pins it (`python.defaultInterpreterPath` → `gscpt/.venv/bin/python`), so opening that workspace auto-selects it. It only matters for `trade`/`transport` (they need pandas); the other scripts are stdlib-only.
- 5.2. IF you ever see "No module named pandas": `Cmd+Shift+P` → "Python: Select Interpreter" → pick `gscpt/.venv/bin/python` (needs the VS Code Python extension). Otherwise, nothing to do.

6. Re 2.5 (deleting `/VS Code`?) —— delete the old venv, NOT the whole folder
- 6.1. `gscpt/` is self-sufficient (scripts + `gscpt/.venv` both live in the repo), so the migrated scripts work with `/VS Code` gone.
- 6.2. The real space win: delete just the OLD `/VS Code/.venv` (602MB) —— it IS the clutter. `gscpt/.venv` is only 117MB.
- 6.3. But do NOT delete the whole `/VS Code` folder yet —— it also holds things that are NOT in the repo: a separate `Temporary Codes/` folder (never migrated), `.vscode/` (editor settings), `requirements.txt`, your `Default.code-workspace`, and the `Culous Codes/` script ORIGINALS (the backups we kept). Sort those first —— especially migrate/relocate `Temporary Codes/` if you still use it.
- 6.4. Why the old venv ballooned to 602MB: over the years it accumulated a full data-science/Jupyter stack —— `ipykernel`, `ipywidgets`, `matplotlib`, `PIL`, `bs4`, `httpx`, and dozens more. Your 8 scripts use only **pandas** (+ its `numpy`/`dateutil`/`six` deps), so `gscpt/.venv` is lean at 117MB —— roughly 485MB of the old one is stuff these scripts never touch.
- 6.5. Heads-up: `Culous Codes/` currently has no bulk input data —— only `202606052208.txt` (likely a leftover DAMF instruction) + `temp.txt`. When you run the data scripts, their INPUT files must sit in `gscpt/` (open confirm 12.5a —— say if you want a `gscpt/data/` subfolder).

7. Re response_202606060508 §1 (upside of root vs `gscpt/`) —— none meaningful for you
- 7.1. A `.code-workspace` behaves identically wherever it lives; placement never limits what it opens. Root's only marginal pluses are discoverability and a few tools that expect a root-level workspace —— neither matters to you, and you wanted it hidden, so `gscpt/GH.code-workspace` is the better fit with no downside.
