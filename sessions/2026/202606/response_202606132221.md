19. Re #12 —— the refinements (all built & tested)
- 19.1. en dash → YELLOW (your default), not RED. I chose yellow over "precise red" because a precise red cannot safely tell a valid ALPHABETIC range (`Jan–Mar`, `Mon–Fri`) from an em-dash substitute, so it would force exactly the "conditional acceptance of a red" you want to forbid. As built: a numeric range `1–2` is silently OK (no flag at all); any other en dash is yellow to judge. Net effect —— RED now only ever fires on a genuine breach, so your zero-tolerance "RED must be 0" holds with no exceptions.
- 19.2. Americanism "corrections" (`color` → `colour`) —— they are NOT auto-applied; dlint only FLAGS, you rewrite. The British form is shown purely as an output hint. Since dlint is run-not-read, those strings cost nothing in context; they appear (briefly) only in the output line when that word is actually flagged. So no long-run bloat —— I kept them.
- 19.3. `-ize`/`-isation` generalisation → YELLOW (your call), with an exceptions guard so `size`/`prize`/`seize`/`maize`/`capsize`/`resize` etc. never fire. Exact words (`learned`, `while`, `color`…) stay RED and unconditional. So the hard rule has no generalisation gap, and the fuzzy rule can't false-block.
- 19.4. Colon —— times (`9:00`) and URLs (`://`) excluded from RED, as you confirmed.
- 19.5. Hart's quotation (§12.3.4) —— built: comma inside a closing quote (`,”`) → RED (it is essentially never original to the quote); period inside (`.”`) → YELLOW (judge whether the full sentence was quoted). CIIW edge case —— the only place `,”` is conventionally correct is fiction-style dialogue ("Wait," she said), which deliverables almost never contain; tell me if you ever need dialogue and I will downgrade it to yellow.
- 19.6. Spaced hyphen (§12.3.1) —— yellow ONLY when a hyphen has a space on BOTH sides (`a - b`); `re-use` and line-start bullets (`- item`) never fire. No gap I can see; the one thing it won't catch is a space-one-side oddity like `word -word`, which is rare and reads as a typo rather than a dash.
