14. Re 9.1 —— yes, and the fix is *better* than you hoped: **per-file SHA**
- 14.1. We do **not** stamp one `HEAD` SHA onto every URL. Instead each file's URL uses *that file's own* last-commit SHA, via `git log -1 --format=%H -- <path>`. The demo just now:
  - 14.1.1. `google.md` → `97ff550`, `glossary.md` → `e103a50`, `numbered.md` → `6a8d569` —— three different SHAs, each frozen at the commit that last touched that file.
- 14.2. Consequence —— exactly what you asked for: a file's URL changes **only when that file changes**. In a `~`100-file CP, if you edited `3`, only `3` URL lines move; the other `97` are byte-identical and I leave them untouched. No mass rewrite, minimal tokens.
- 14.3. Detection is pure git, not content: the script computes each file's current last-commit SHA and compares it to the SHA already in `index_otg.md`; it edits only the differing lines and prints a short "changed: x, y, z" summary. I never read file bodies, so token cost ≈ the summary.
- 14.4. **Your mtime fear cannot happen.** The mechanism reads git history; it never opens, edits, or `touch`es any file. So "I didn't edit it but its mtime became now" is structurally impossible in `#sync`.
  - 14.4.1. The `google.md` mtime jumping to `02:02` was the one-off **manual sentinel** I added then reverted for the trial (a local Edit, twice) —— not the mechanism. I've just restored its mtime to its true last-commit time, `2026-05-30 07:35:28`. Verify in Finder/GH desktop.
  - 14.4.2. Bonus: `git log -1 -- <file>` is actually a *more* trustworthy "last modified" than mtime —— it survives copies, syncs and clones, which mtime doesn't. So you gain a better history, not lose one.
