2. How I read your intent
- 2.1. You're porting a CWI-era device (each role = a static artefact, role-played sequentially by one model) onto CC's real affordance: genuinely independent, parallel agents that react to each other through a shared file.
- 2.2. The board is meant to be the single source of truth and cross-role memory; append-only exists to stop concurrent writers clobbering each other.
- 2.3. `THE_END` + polling is your attempt to keep SAs "live" so the debate flows in real time rather than in lockstep.

3. Where I fully agree
- 3.1. The CWI→CC re-architecture is correct and overdue —— artefacts were a workaround; parallel SAs are the genuine article.
- 3.2. Shared board as source of truth: right call.
- 3.3. Append-only instinct: right diagnosis —— Edit/Write would clobber concurrent entries.
- 3.4. Retaining a degraded Non-CC path: sensible and cheap.

4. My main concern —— the execution model (the crux)
- 4.1. "SAs wake every minute until they see `THE_END`" assumes long-lived, self-polling agents. CC SAs don't behave that way: an SA spawned by MA runs once and returns, whilst MA blocks awaiting that return.
- 4.2. That is a deadlock: MA can't judge saturation and write `THE_END` until the SAs return —— but the SAs won't return until they see `THE_END`. Each side waits on the other.
- 4.3. Separately, foreground `sleep` is blocked in this harness, so "wake every minute" can't be a naive sleep-loop regardless —— it would need background tasks + MA polling the board, which is token-heavy and fragile.
- 4.4. My recommendation —— round-based orchestration, not self-polling:
  - 4.4.1. MA spawns all role-SAs in parallel (one message, N Agent calls) for Round 1. Each SA: reads the board → appends ONE argument block → returns a one-line ack.
  - 4.4.2. MA reads the board, judges saturation. Not saturated → spawn Round 2 (same roles, now seeing all prior rounds). Saturated → append `THE_END` + Observer.
  - 4.4.3. This keeps your parallelism AND your shared board, kills the deadlock, and gives MA clean control of rounds and termination. The `THE_END`/poll machinery then becomes unnecessary.
- 4.5. If you genuinely want emergent real-time exchange, the only sound version is background SAs + MA polling the board —— I can spec it, but it's materially costlier and flakier. I'd default to 4.4.

5. Secondary issues to fix in Task 2
- 5.1. Append safety isn't fully solved by `>>`. Concurrent appends over `~`4KB (macOS PIPE_BUF) can still interleave mid-block. Two options:
  - 5.1.1. Keep `>>` but mandate one block per `printf` and keep blocks small.
  - 5.1.2. Cleaner: each SA writes its block to its OWN file (`debate_[TS]_[role].part`); MA merges into the board with round headers. Zero shared-file contention, and MA controls ordering. I lean 5.1.2.
- 5.2. Output surfacing —— the board is a working file, but you only ever read `response_`. The file as written never says how the verdict reaches you. Task 2 must mandate: the Observer verdict (+ scores) is distilled into `response_[TS].md`; the board stays as the audit trail.
- 5.3. Observer placement is ambiguous (same board vs separate). I'd make Observer a final independent SA (your unincentivised-McKinsey framing), writing its block to the board, after which MA lifts the verdict into `response_`.
- 5.4. SA briefing is unspecified. Task 2 should add a fixed briefing template each SA receives: topic/decision, assigned role + stance, who it opposes, "read board first", the append rule, the Debater block format, and the stop condition.
- 5.5. Caps —— add sane defaults: role count, max rounds, and SA model (default Sonnet; Opus only where the decision warrants it, per c.md §2.9.4). Stops a debate ballooning in cost.
- 5.6. Creation vs append —— clarify MA may Write/heredoc to CREATE the board; the append-only rule governs additions thereafter.
