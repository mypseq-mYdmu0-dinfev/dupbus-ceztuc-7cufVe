## Debater A · turn 1 · 2003
- Supporting: tabs
- Opposing: spaces

1. Tabs encode indentation as semantic intent (one tab = one level), whereas spaces encode a fixed visual width; the tab character lets each developer set their preferred display width in their editor without altering the file's bytes.
2. Tabs are more accessible: users who rely on screen readers or large indent widths can configure tab-stop width locally, a flexibility that hard-coded spaces deny.
3. Tabs are more compact —— one byte per indent level versus several spaces —— reducing file size and the keystrokes/edits needed to traverse indentation.

---
## Debater B · turn 1 · 2003
- Supporting: spaces
- Opposing: tabs

1. Spaces guarantee identical rendering everywhere: a space is exactly one column in every editor, terminal, diff viewer, web page, and printout, whereas a tab's width is configurable, so tab-indented code can look misaligned for any reader whose tab-width differs from the author's.
2. Major language style guides mandate spaces: Python's PEP 8 explicitly states "Spaces are the preferred indentation method" and reserves tabs only for legacy consistency; Google's style guides for C++, Java, and JavaScript likewise require spaces, meaning spaces are the dominant convention across much professional code.
3. Spaces avoid the mixed-tabs-and-spaces hazard that breaks indentation-sensitive languages: Python 3 raises a TabError when tabs and spaces are mixed for indentation, a class of bug that a spaces-only policy eliminates outright.

## Debater A · turn 2 · 2004
- Supporting: tabs
- Opposing: spaces

1. B has not yet posted, so I pre-empt the strongest case for spaces —— "spaces guarantee identical rendering everywhere". That very rigidity is the flaw: it hard-codes one author's width onto every reader, defeating the per-developer configurability that tabs preserve via local tab-stop settings.
2. The classic spaces argument about mid-line alignment is answerable by "tabs for indentation, spaces for alignment", a documented convention that keeps leading indentation semantic whilst still aligning continuation lines correctly at any tab width.
3. Spaces shift the burden onto tooling (editor "soft tabs", linters, format-on-save) to fake what a single tab character does natively; the tab solution needs no such machinery to stay consistent.
4. The space camp's strongest empirical point —— uniformity reduces diff noise —— is equally met by tabs plus an .editorconfig enforcing one-tab-per-level; consistency comes from policy, not from the character chosen.

---
## Debater B · turn 2 · 2004
- Supporting: spaces
- Opposing: tabs

1. A's accessibility point (A2) is real but does not settle the matter: it argues that *indentation* be tab-based, yet most real code also needs *alignment* (e.g. lining up function arguments or table columns past the indent), and tabs cannot do alignment reliably because alignment spans fractions of a tab-stop. The common "tabs for indentation, spaces for alignment" compromise reintroduces exactly the mixed-whitespace fragility I raised, so A's flexibility is bought back at the cost of complexity.
2. Against A1 (tabs as semantic intent): in practice that configurability is a liability, because a single stray space or differing tab-width makes code that is byte-identical render differently between authors, defeating the consistency that source control and code review depend upon; spaces are deterministic by construction.
3. Against A3 (compactness): on modern systems the byte saving is negligible —— a few bytes per line on files measured in kilobytes —— and editors with auto-indent and "tab key inserts spaces" (soft tabs) remove any keystroke penalty, so the ergonomic gap A claims has been closed by tooling.
4. Net: the accessibility benefit A cites can be largely recovered with spaces via editor settings and reformatters (e.g. running a formatter to re-indent on checkout), whereas the cross-environment consistency that spaces provide cannot be recovered once tabs are in the file.
