26. Re 8 / 8.1 —— the precise mechanism, and CIIW on "twice":
- 26.1. Git stores no renames; `git log --follow` reconstructs them with **no hop limit** —— so on the CLI history is never lost, however many moves.
- 26.2. GitHub's **web** path-History (`/commits/<branch>/<path>`) does NOT `--follow` —— so it shows history only since the file last arrived at the current path. Strictly, it breaks at the **first** rename, not the second. What lets you go further is **"View blame → blame prior to this change"**, stepping back one rename at a time (unlimited but manual). So your "more than twice = gone" is really "the clean History list breaks at every rename; blame-stepping is the manual recovery, and it gets tedious fast." (CIIW: there isn't a hard 2-hop cutoff —— it's 1 for the list, N-but-painful for blame.)
- 26.3. The procedure you asked for —— "find out if it's been renamed already" —— I built + ran it (read-only): `git log --follow --diff-filter=R --name-status -- <file>` lists every `R<score> old new` hop. We run this BEFORE any move to see each file's hop count and similarity scores.

- 27.2. Your "stupid way" (recreate file, replay each version) is the manual, lossy, un-backdatable version of a tool that does it **properly and automatically**: **`git filter-repo --path-rename`**. It rewrites history so the file *was always* at the new path —— zero hops consumed, full unbroken GH-UI history with red/green diffs. That is the smart way you were reaching for.
- 27.3. Two grades of it: (a) `--path-rename dissertation/:cp/dissertation/` fixes only the *folder move* (the pre-existing internal chains like the 4-hop file remain); (b) a **full linearisation** —— map every historical name of each file → its final `cp/dissertation/…` path (scriptable from the `--follow` audit) —— gives a perfect single-path history for ALL dissertation files.
