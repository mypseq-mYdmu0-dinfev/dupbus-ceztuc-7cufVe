## Issue 2 —— MS-Office "Grant File Access" alerts (evaluated; need 1 decision)

3. What that dialogue actually is —— I read the screenshot + `document.wflow`. It is **not** a macOS TCC/Full-Disk-Access prompt. It is **Microsoft's own** sandbox "powerbox" gate: when an external script (`tell application "Microsoft Word" … open file …`) asks a *sandboxed* Office app to open a file for which it holds no security-scoped bookmark, Microsoft shows "Grant File Access" and makes you pick the item. It fires **per file**.

4. Why iWork is silent but MS isn't —— your converter drives iWork via `Pages/Numbers/Keynote Creator Studio` (non-sandboxed builds; they export straight to PDF with no gate), whereas Word/Excel/PowerPoint are App-Sandboxed and enforce this per-file grant. Same automation, two different sandbox regimes.

5. Three ways to kill the alerts (ranked):
- 5.1. **Option C —— LibreOffice headless (best; truly hands-off).** Swap only the MS branch of the AppleScript for `soffice --headless --convert-to pdf --outdir <ts-folder> <file>`. No GUI, no sandbox, no prompt, and Office need not even be running. Handles `.docx/.doc/.xlsx/.xls/.pptx/.ppt`. One-time `brew install --cask libreoffice` (`~`700MB; brew is present, LibreOffice is not). iWork branch stays exactly as-is. Fidelity is high; rare minor layout drift vs Word.
- 5.2. **Option B —— grant the parent folder once per app (no new software).** When the dialogue appears, click `Select…`, navigate UP, and choose the **`PDF Conversion`** folder (not the file). Office stores a scoped bookmark and stops prompting for that folder. Caveat: per-app (Word, Excel, PPT each need it once) **and** your files live on an external volume (`/Volumes/FURY 2TB`) —— Office bookmarks for external volumes are fragile and can need re-granting after a remount. So "grant all" works but isn't bullet-proof here.
- 5.3. **Option A —— Full Disk Access to the Office apps —— won't reliably work.** FDA is a TCC permission; the App Sandbox is a stricter, separate kernel layer that FDA does not widen. The "Grant File Access" dialogue is Microsoft's bookmark gate, not TCC, so FDA generally does **not** suppress it. I'd skip it.

6. My recommendation —— **Option C**, precisely because your files sit on the external FURY volume where Option B's bookmarks are the most fragile. It's the only one that makes the automation "truly run by itself" indefinitely. I did **not** install LibreOffice or rewrite the `.wflow` unilaterally —— it's a new `~`700MB dependency + a converter rewrite, your call first.

8. Your naming question —— what you're preserving is **history continuity** across renames (umbrella term). The mechanism words:
- 8.1. Git stores **no** renames; it *infers* them via **rename detection** (content-similarity heuristic, default `~`50%). `git log --follow <file>` is what walks across a rename; the GitHub web "History/Commits-for-path" view does **not** use `--follow`, which is exactly why your chain visibly dies at a rename.
- 8.2. The technique that maximises it: a **pure rename** —— a commit that renames and changes **nothing** else → Git records `R100` (100% similarity), which every tool can follow. Bundling a move with edits in one commit can drop similarity below threshold and *break* detection.
- 8.3. So your phrasing becomes: *"move [file] as a pure rename, preserving its **history continuity** (keep it `--follow`-able / `R100`)"*. Lead with "history continuity"; say "rename-following" if you want the mechanism.
- 8.4. Honest caveat —— the GH **web** History limitation can't be fully eliminated (GH just doesn't `--follow` there). But pure renames keep every hop detectable, so you can still step back via **"View blame prior to this change"**, and `git log --follow` always recovers the full lineage locally.

- 16.1.
