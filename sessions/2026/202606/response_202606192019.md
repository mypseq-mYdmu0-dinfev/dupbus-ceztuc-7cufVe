# Response to query_202606192019.md

1. Yes, this works — `/loop` stays in the existing session, unlike `/schedule`.

2. Your proposed flow is valid, with one tweak for precision:
- 2.1. `/loop 15m` fires ~15 min after it starts, so if you send it at 00:00:30, it wakes at 00:15:30 — close but not exact
- 2.2. For tighter precision, use a shorter poll: `/loop 1m` + I check `date` each wake and act only when time >= target

3. Recommended flow for "do X at 00:15":
- 3.1. Msg 1: describe the task + target time (`00:15 SYD`)
- 3.2. Msg 2: `/loop 1m` — I check the clock each wake, execute once at 00:15, then stop the loop

4. Caveats:
- 4.1. App must stay open (same constraint as `/schedule`)
- 4.2. Tell me to stop after executing, or I'll keep waking past the target time
