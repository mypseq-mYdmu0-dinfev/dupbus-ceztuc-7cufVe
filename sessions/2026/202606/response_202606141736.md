# Response to query_202606141736.md
*Modes, coverage extension, deliverable-hook caveat, OTGC*

1. Diff between modes
- 1.1. There are three invocations, but only TWO rule-sets:
  - 1.1.1. FULL file —— `python3 cscpt/dlint.py <path>`: runs EVERY rule and auto-fixes quotes IN PLACE. For a separate-file deliverable.
  - 1.1.2. FULL `--text` —— `--text "..."`: same full rules on a string; prints the quote-fixed text instead of writing a file. For a deliverable embedded in a `response_`.
  - 1.1.3. `--quick` (works with either a path or `--text`): runs ONLY the register-independent subset and NEVER rewrites the file.
- 1.2. What `--quick` drops vs FULL —— it keeps Americanisms (RED), Hart's quotation (RED/YELLOW), `-ize` (YELLOW) and the hyphen/#numbered check (YELLOW); it SKIPS em dash, mid-sentence colon, en dash, bare `+`, the GenAI word + phrase scan, and sentence-initial `Where`. Those skipped ones are either deliverable-only (em dash, colons are fine in our internal comms) or heavy/heuristic (the GenAI list).
- 1.3. How is it "quicker"? Mostly by doing LESS work —— it skips the GenAI scan, which is by far the largest (a `~`90-word set plus phrase substrings across every line). In wall-clock terms both modes are still sub-100ms, so the honest answer is that "quick" really denotes the LIGHTWEIGHT, comms-SAFE ruleset, not a meaningful speed gain. The name is a slight misnomer.
- 1.4. Tokens/context —— neither mode loads the script (run-not-read), so the script itself costs nothing either way. The only context cost is the printed OUTPUT. `--quick` runs fewer rules, so on a FLAGGED file it prints fewer flag lines and thus burns marginally fewer tokens; on a CLEAN file both print the same `~`6 lines. So the token saving is real but minor, and only via shorter output.

2. Coverage —— `close_`, CP prefixes, extension
- 2.1. The `close_` lint was MY OWN intention, not the hook. The hook fires only on a filename containing `response_`, so `close_` did not trigger it; I ran `--quick` on it voluntarily because #numbered/Hart matter in comms I author.
- 2.2. CP prefix —— good news: the hook ALREADY covers CP sessions. It keys on the SUBSTRING `response_`, and a CP file is `career_response_[TS].md`, which still CONTAINS `response_`. So CP-prefixed `response_` files are caught. (It deliberately ignores `query_`, since those are your words, even when I author a QMM `query_`.)
- 2.3. Extending to `close_` + `wrap_` —— I agree; they are CC-authored comms where the same rules apply. Mechanically it is a small change to BOTH the `.sh` fast-path (match `response_`/`close_`/`wrap_`) and `.py` (basename contains any of the three). I would NOT add `query_` (user content) or `artefact_` (CWI/OTGC-generated, not local). Ready to do this when you say go.

3. Deliverables hook —— appealing, but one factual correction changes it
- 3.1. The premise "the 1% system files would only get YELLOW flags" is unfortunately wrong. A deliverable is linted in FULL mode, and FULL mode RED-flags em dash and mid-sentence colons —— which our INTERNAL/governance `.md` files (root `CLAUDE.md`, `writing.md`, `glossary.md`, the `README.md`s, etc.) use legitimately and heavily (em dash is allowed internally per §2.4). So a rule of "any non-comms `.md` → FULL lint" would RED-BLOCK routine edits to our own governance files, not merely yellow-flag them.
- 3.2. Your `.txt` nuclear override only helps NEW files; editing an EXISTING governance `.md` (e.g. `CLAUDE.md`) still triggers the hook on the `.md` write and blocks on its em dashes. So that escape hatch does not cover the common case.
- 3.3. The real discriminator is LOCATION, not extension. Deliverable separate-files live in `/sessions/...` (§3.7.2); governance files live in `/universal/`, root, `/cscpt/`, etc. So a safer version of your idea: FULL-lint a `.md` in `/sessions/` that is NOT `query_`/`response_`/`close_`/`wrap_`/`artefact_`. That excludes governance files (wrong location) and comms (wrong name), leaving genuine deliverable files. A deliverable should have no em dash anyway, so a RED there is CORRECT.
- 3.4. Residual caveats of the location-scoped version —— a rare hand-written scratch `.md` in `/sessions/` could RED-flag (mitigable by naming it `temp_`), and non-`.md` deliverables (`.docx` via a skill, text pasted into Pages) still cannot be hooked (FULL dlint cannot read a binary anyway). So it tightens the net but is not total.
- 3.5. Recommendation —— do not adopt the plain `.md`-pattern; if you want a deliverable hook, use the `/sessions/`-scoped version in 3.3. Happy to build it, but let us confirm the scope first since a mis-scoped FULL hook is disruptive.

4. OTGC compatibility
- 4.1. Can OTGC run `dlint.py` if its URL sits in `index_otg.md`? No. OTGC (and CWI) have NO code execution / shell at all; a URL only lets her FETCH and READ text, never EXECUTE Python. So the URL does not enable running.
- 4.2. Since she cannot run it —— yes, the right move is a READABLE rules companion for non-CC, pointed to by `writing.md` for CWI/OTGC. Non-CC simply applies the rules by hand. That is strictly better than shipping Python she cannot execute.
- 4.3. The "can run / can run without reading" branches are therefore N/A —— there is no execution path on OTGC.
- 4.4. A `dlint_otg.py` would NOT help —— reading Python in order to "run" it still needs an interpreter she lacks; and reading code to mentally emulate it is far worse than reading plain-prose rules. So the lightweight artefact for non-CC should be a RULES `.md`, not a script.
- 4.5. Suggested shape (not acting) —— a concise `universal/dlint_rules.md` (or a clearly-marked non-CC section) holding the human-readable RED/YELLOW rules plus the banned-word list, mirroring `dlint.py`. Trade-off: it duplicates the list, so it can drift; mitigate by treating `dlint.py` as source of truth and refreshing the `.md` whenever the list grows. If you prefer zero duplication, non-CC can instead lean on `writing.md`'s illustrative list plus judgement, accepting it is non-deterministic.
- 4.6. My recommendation —— defer unless you actually lint deliverables on OTGC often; if you do, the readable `.md` (4.5) is the way, not a Python variant.
