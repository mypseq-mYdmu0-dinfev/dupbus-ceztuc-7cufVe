# Response to query_202606030103.md
*OTG staleness: bust the URL, not duplicate the file*

1. Diagnosis —— confirmed, and it's a cache problem, not a GH problem
- 1.1. There are **two** cache layers between you (OTG/CWI) and the file, both keyed by the **exact URL string**:
  - 1.1.1. GitHub raw CDN —— ~`5` min TTL nominally, but documented to serve content hours (even `~`48 h) stale ([GH discussion #46758](https://github.com/orgs/community/discussions/46758), [#188334](https://github.com/orgs/community/discussions/188334))
  - 1.1.2. Claude's own web-fetch cache —— ~`15` min TTL per URL; on a hit it returns the cached body without re-fetching ([Claude web-fetch docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/web-fetch-tool), [analysis](https://mikhail.io/2025/10/claude-code-web-tools/))
- 1.2. Why opening it yourself shows latest: your browser request isn't behind Claude's fetch cache, and it likely landed after the CDN had refreshed. So the stale "ver 01" you saw is the fetch-layer cache, exactly as you suspected.
- 1.3. **The key realisation**: your proven versioned-duplicate trick works for one reason only —— renaming `google.md`→`google_05.md` changes the **URL**, so both caches treat it as a brand-new resource and miss. The file duplication is incidental; the URL change is what does the work.

2. So the fix is to change the URL without duplicating the file
- 2.1. Two clean ways to make the URL unique per version, neither needing a single duplicate file:
  - 2.1.1. **Commit-SHA permalink** (recommended): `.../dupbus-ceztuc-7cufVe/<commitSHA>/universal/google.md` instead of `.../main/universal/google.md`
  - 2.1.2. **Cache-bust query param**: `.../main/universal/google.md?v=<TS>` —— appending a unique query string forces a miss on both layers ([cache-busting refs](https://github.com/orgs/community/discussions/169198))
- 2.2. I recommend 2.1.1 (permalink). It is strictly better than 2.1.2:
  - 2.2.1. A SHA URL is **content-addressed & immutable** —— that exact URL can *never* go stale, so Claude caching it for `15` min becomes correct behaviour, not a bug. The query-param route still relies on the CDN honouring query strings (usually yes, occasionally stripped).
  - 2.2.2. No timestamp guesswork; the SHA *is* the version identifier.
  - 2.2.3. Your repo is **public** (I confirmed `raw...main/glossary.md` returns `200` with no token), so permalinks resolve with no auth.

- 4.2. Solves the index's own chicken-and-egg: the index is cached too, so it also needs a fresh URL. Because you re-paste userPref each session anyway, dropping the index's new permalink there each run keeps it fresh with no extra step.
