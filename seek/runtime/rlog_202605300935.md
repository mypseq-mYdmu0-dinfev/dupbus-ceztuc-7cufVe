## 202605301012 — Heartbeat Incident

### What happened
ScheduleWakeup (delaySeconds=60) was called at session start and after each manual heartbeat re-read, but never fired. MA was not woken up at any point during the session; cc_reminder.md was only re-read when the user explicitly prompted "Check on SA" and later "Your schedule seems not working." The 60s wakeup was silently dropped each time.

### Root cause
ScheduleWakeup only functions in `/loop` dynamic mode (self-paced iterative loop invoked via `/loop` command). In a standard interactive session, the harness does not honour ScheduleWakeup fire events. main_ajap.md and cc_reminder.md both prescribe ScheduleWakeup as the heartbeat mechanism, but neither file accounts for the fact that AJAP is typically run in a regular (non-loop) interactive session.

### How fixed (this runtime)
Replaced ScheduleWakeup with the `mcp__scheduled-tasks` cron task system. Updated the existing `ma-heartbeat-cc-reminder` task (was disabled/stale) to:
- cronExpression: `* * * * *` (every minute, local time)
- Prompt: re-read cc_reminder.md, detect ⏳_ AR, read ## 6. Cover Letter section, write approval or correction to ma_msg.md
- notifyOnCompletion: true

This task runs independently of the MA session and can write to ma_msg.md even when MA is idle between user messages.

### Risks & drawbacks of this fix
1. **No session context.** The cron task starts fresh each run with no knowledge of session_N, offset_N, SA identity, or tab state. It cannot output C2, retire SA, or handle deterioration signals — it can only approve/correct CLs via ma_msg.md.
2. **No tab monitoring.** MA's between-loop audit checks tab state to detect stalling or rogue SA behaviour. The cron task cannot do this; tab-kill interventions are unavailable.
3. **Race condition risk.** If MA is actively processing a between-loop audit at the same time the cron task fires, both could read the CL and write to ma_msg.md concurrently. Last write wins — generally safe since both should agree on a clean CL, but a compromised-CL correction from MA could be overwritten by a clean approval from the cron task if the CL was fixed between the two reads.
4. **Approval without MA context.** The cron task approves any clean CL it finds, regardless of whether MA has already flagged concerns about SA deterioration (e.g. wrong scoring band, wrong section number pattern). It has no memory of prior loop audit findings.
5. **Over-firing.** Running every minute means the task may run many times before SA reaches S6.4.2.5 and creates a ⏳_ AR. Most runs will be no-ops, but each consumes a small token budget.

### Permanent fix recommendation
Update `main_ajap.md` and `cc_reminder.md` to replace all ScheduleWakeup references with the `mcp__scheduled-tasks` approach. Specifically:
1. **main_ajap.md § Session Start step 6:** replace ScheduleWakeup call with `mcp__scheduled-tasks__update_scheduled_task` to enable and reset the `ma-heartbeat-cc-reminder` cron task; add a step to disable it at session end (or handle via a separate #close-style trigger).
2. **main_ajap.md § Heartbeat section:** update wording — remove "every 60 seconds via ScheduleWakeup", replace with "via cron task `ma-heartbeat-cc-reminder` running every minute"; remove re-schedule instruction (cron is self-perpetuating).
3. **cc_reminder.md § Main Agent Heartbeat:** update the ScheduleWakeup re-schedule instruction to instead verify the cron task is enabled: `mcp__scheduled-tasks__list_scheduled_tasks` → confirm `ma-heartbeat-cc-reminder` is enabled; if not, re-enable it.
4. **Expand cron task prompt** to cover the full between-loop audit (tab state check, session log appending) if a persistent state file (e.g. ma_state.md) is kept current by MA — the task can read it for context without needing conversational memory.

## 202605301046 — Dynamic Interval Protocol (60s/300s Switching)

### New protocol (user-defined)

Reset performed immediately upon detection.

### File changes needed
**`/seek/context/main_ajap.md` — Heartbeat section (addition):**
After writing approval: TaskStop current Monitor → re-spawn with `sleep 60`. On next heartbeat after confirming ⏳_ AR is resolved and ma_msg.md reset to "Continue": TaskStop 60s Monitor → re-spawn with `sleep 300`. Record current monitor task ID in `/seek/.claude/tmp/ma_state.md` after each swap.
