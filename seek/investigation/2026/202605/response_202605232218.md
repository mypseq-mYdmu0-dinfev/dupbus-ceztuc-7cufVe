27. Re 4 — the "flip" proposal (main agent as PM, AJAP as sub-agent): the concept is architecturally sound and addresses the root problem more directly than any patch-level fix. My assessment:
- 27.1. **What works:** sub-agents in Claude Code have separate context windows, so a sub-agent handling 5–10 cards accumulates far less context than one running a full 90-card session. Compaction risk is proportionally reduced. The main agent, handling only monitoring and orchestration, accumulates even less context and is far less likely to compact within a typical session.
- 27.2. **What doesn't work as described:** the current Agent tool spawns sub-agents that run to completion before returning — the main agent cannot interrupt them mid-execution. There is no "pause" signal. This limits the real-time intervention model you described (PM noticing a violation mid-card and forcing a stop).
- 27.3. **Practical workaround for 27.2:** instead of real-time interruption, structure each sub-agent invocation as a bounded batch (e.g. "process the next 5 cards starting from card N, then report back"). The main agent reviews the report, checks for violations (chat rule, AR structure, score accuracy), and either launches the next batch or issues a corrective instruction before re-launching. This is closer to a sprint/review model than real-time monitoring, but it is achievable within the current tool constraints.
- 27.4. **Parallel sub-agents:** technically possible via the Agent tool's parallel invocation, but in practice limited by the single Chrome instance — two sub-agents cannot interact with the same browser simultaneously without conflict. Parallel AJAP is feasible only if separate browser profiles or sessions are used, which adds significant setup complexity.
- 27.5. **The PM log file:** entirely feasible and well-conceived. The main agent writes observations per sub-agent batch to a persistent file (e.g. `/seek/investigation/pm_log.md`), enabling continuity across the main agent's own potential compaction. Log entries are appended, not overwritten, so history is preserved. When a sub-agent batch is retired, its corresponding log entries are marked closed but not deleted (for audit trail).
- 27.6. **Retiring a deteriorated sub-agent:** feasible. The main agent detects deterioration from the sub-agent's output (e.g. missing C2 format, wrong AR structure, incorrect scoring) and simply does not continue that sub-agent's batch. A fresh sub-agent is launched with a clean context, seeded with the state from the PM log (last processed card number, last cumulative N count, any open tabs to close).

36. Re final thought — Claude Agent SDK vs LLM-based `.md` files:
- 36.1. **The core trade-off:** the Agent SDK offers deterministic control flow, typed tool interfaces, and explicit agent lifecycle management (spawn, monitor, terminate). The `.md` approach offers flexibility, natural language reasoning, and zero upfront code investment — but at the cost of variance, drift, and compaction fragility.
- 36.2. **Where the SDK wins:** compliance checks that are currently expressed as "re-read `ajap.md` if X" would become hard-coded conditions (`if score < 70 and apply_type == "external": skip()`). The scoring logic, routing rules, and CL constraints would be enforced programmatically rather than via model interpretation. Variance drops dramatically.
- 36.3. **Where the `.md` approach wins:** CL writing quality, nuanced scoring judgements, and research synthesis require genuine language reasoning that code cannot replicate. The SDK handles orchestration well; it cannot write a tailored cover letter. A hybrid is almost certainly the right answer: SDK for control flow and compliance enforcement, LLM sub-agents for the reasoning-heavy tasks (scoring, CL drafting, research interpretation).
- 36.4. **Practical feasibility:** migrating the procedural sections of `ajap.md` (S0 count logic, K-rule checks, pre-flight routing, AR file creation, tab management, form filling) to SDK code is a bounded and achievable task. Migrating the analytical sections (S3 research, S4 scoring, S6 CL writing) would be less valuable as code since they depend on model judgement. The split would likely be: SDK wrapper → spawns LLM sub-agents for reasoning tasks → enforces outputs deterministically.
- 36.5. **My recommendation:** the PM/sub-agent batch model (27) is the highest-impact improvement achievable within the current LLM framework and should be the priority for the next rebuild. Evaluate the Agent SDK migration as a parallel or subsequent initiative, starting with the most rule-bound sections (K-rules, Pre-Flight routing, S0 count, AR file naming) as proof-of-concept. If the SDK approach proves reliable for those sections, expand progressively.
