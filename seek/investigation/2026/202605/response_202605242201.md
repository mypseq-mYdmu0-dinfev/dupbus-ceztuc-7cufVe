## Problem 1 — SendMessage / SA Continuation Failure

1. **Your understanding is correct.**
- 1.1. The Agent tool description states: *"To continue a previously spawned agent, use SendMessage with the agent's ID or name as the `to` field."* However, the actual tool schema has no `to` field — only `description`, `isolation`, `model`, `prompt`, `run_in_background`, and `subagent_type`. SendMessage as a SA continuation mechanism **does not exist** in this runtime environment.
- 1.2. Every MA "continue" therefore spawns a **fresh SA** — which re-reads all 6 mandatory files from scratch — instead of resuming the existing SA mid-session.
- 1.3. Net effect: each card costs ~6 mandatory-file reads rather than amortising them across the session. On a 100-card session this is an enormous token overhead.

2. **Proposed solutions (not ranked; your call on approach):**
- 2.1. **Wait for runtime support (no code change needed):** if the Claude Code harness eventually ships `to`-field support on Agent, the original design works as-is. Low effort; uncertain timeline.
- 2.2. **Redesign SA scope — run until a natural stop point (recommended):** instead of stopping after every card, SA should continue processing cards autonomously and only stop when:
  - It reaches S6.4.2.5 (needs MA CL approval before Submit), OR
  - A flag requiring MA judgement arises (e.g. ambiguous Consultant routing, struggling portal), OR
  - It has completed a configurable batch ceiling (e.g. 10 cards) as a safety valve against runaway processing
  - This way mandatory file reads are amortised across ~10 cards per spawn, not 1. The "one card only" instruction that was added mid-session was an overcorrection to a runaway-batch problem — it cured that symptom but amplified the token cost.
- 2.3. **Lightweight "warm start" file (optional complement to 2.2):** MA writes a compact state snapshot to a known tmp path after each loop (e.g. `/tmp/ajap_sa_context.md` — current N, last AR path, any active flags). Fresh SA reads this single file first, then reads mandatory files only if a version/hash marker indicates they have changed. More complex to implement; marginal benefit if 2.2 already reduces spawn frequency.
- 2.4. **What NOT to do:** do not try to replicate SendMessage via creative prompt engineering (e.g. "imagine you are the SA from previous turn") — this yields hallucinated continuity, which is worse than a clean re-read.

3. **Recommended path:** implement 2.2 first (adjust SA batch scope in `ajap.md` and SA spawn prompt in `main_ajap.md`). If token cost is still unacceptable after that, layer in 2.3.

## Problem 2 — `ma_state.md` and Session Reporting

4. **Your design intent is sound; what actually happened diverged from it in two ways:**
- 4.1. `ma_state.md` became an operational state file (tabs, last N, clean/dirty state) — useful for MA continuity across heartbeat re-invocations and compaction recovery, but **not** a session report.
- 4.2. No session report file was ever created. Key incidents (SendMessage failure, rogue tab closures, recurring dash violations) were surfaced in chat during the session but never written to a persistent, reviewable record.

5. **On `ma_state.md` specifically:**
- 5.1. Agree it belongs in `/tmp/` (e.g. `/tmp/ajap_ma_state.md`) — it is per-session ephemeral state, not a lasting artefact. The `/seek/runtime/` folder should be reserved for records you, as the CEO, actually want to keep.
- 5.2. It is already wiped at session start (`rm -f ...`), confirming its tmp nature. Moving it just makes that explicit.

6. **On the session report (`ma_[timestamp].md` in `/seek/runtime/`):**
- 6.1. The PM-to-CEO framing is the right mental model: surface only what requires your attention or action — not a card-by-card ledger (you have ARs for that).
- 6.2. Suggested report structure:
  - 6.2.1. **Session header:** date, start N (resumed from), end N, cards processed this session, applied count, skipped count, pending count — one line of numbers
  - 6.2.2. **Infrastructure issues:** tool/capability failures (e.g. "SendMessage unavailable — all continuations spawned fresh SAs; estimated +X token overhead"), unexpected tab behaviour (e.g. "SA closed all tabs at N=42 — unknown cause")
  - 6.2.3. **Systemic compliance failures:** patterns that recurred despite corrections (e.g. "em dash `—` violated in 3 of 5 CLs before SA retirement; persists across different SA instances — likely a training-data pattern"), newly discovered banned-word derivatives
  - 6.2.4. **Routing/rule edge cases:** cases where a rule was ambiguous or applied incorrectly, worth refining in `ajap.md` (e.g. "K6 bypassed ❌_-prefixed voided ARs — silent skip of already-voided entries")
  - 6.2.5. **Pending actions for you:** items requiring CEO decision that MA could not resolve (e.g. jobs in `/pending/` flagged ⭐❗, struggling portals needing manual answers)
- 6.3. What the report should **not** contain: per-card skip reasons, normal-flow events, anything already captured in ARs.
- 6.4. The MA should write this file **incrementally** (append key incidents as they occur, not reconstruct at session end) — session end is often a rate limit or crash, so reconstruction would be lossy.

- 7.1. `main_ajap.md § Session Start` → change `ma_state.md` path to `/tmp/ajap_ma_state.md`; add step to create `ma_[timestamp].md` in `/seek/runtime/` at session start
