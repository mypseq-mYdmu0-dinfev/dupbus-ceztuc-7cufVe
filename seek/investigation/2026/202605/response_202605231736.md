4. Issue 2.3 and sub-agent proposals: sub-agents in Claude Code have real limitations — they are spawned as separate agent calls, cannot directly interrupt or "pause" the main agent mid-tool-call, and consume tokens independently. Background sub-agents can signal via chat text but cannot guarantee the main agent reads that signal promptly if it is mid-loop. With those limitations in mind:
- 4.1. **Sub-agent 1 (10-min runtime heartbeat):** architecturally feasible as a background `ScheduleWakeup` call at session initialisation — fire every 10 min → main agent re-reads `cc_reminder.md` → continues. However, `ScheduleWakeup` fires a new top-level invocation rather than interrupting mid-loop, so it cannot force a pause; it can only inject a reminder at the next natural turn boundary. Recommend adding to `§ Session Initialisation`: `ScheduleWakeup(delaySeconds=600, prompt="Re-read cc_reminder.md now per S0.1, then continue the loop from where you left off.", reason="10-min cc_reminder.md enforcement heartbeat")` — and the prompt must re-schedule itself each firing to continue.
- 4.2. **Simpler alternative to sub-agents (recommended):** instead of sub-agents, add a deterministic trigger inside `cc_reminder.md` itself: "if you have processed more than 3 cards since last reading this file from disk (not memory), re-read `ajap.md` immediately." This is stateless and does not require parallelism or scheduling.

5. Issue 2.4: sub-agents 2 and 3 (per-portal and per-page timers):
- 5.1. **Sub-agent 2 (entire portal ops timer):** feasible as a background `ScheduleWakeup` launched upon entering an external portal — fire once after the configured limit (≤10 min from `ajap.md`) → check if still on the same employer/role → treat as struggling if so. This is more reliable than a sub-agent because it requires no parallelism; just a single deferred fire.
- 5.2. **Sub-agent 3 (per-page timer):** more complex because it must loop per page change. A `ScheduleWakeup` that re-fires every 3 min and checks URL/title could work but is expensive. Recommended alternative: enforce the ≤3 min page limit via the existing struggling rule by adding: "if you notice you have taken more than 3 tool calls on the same external portal page without progressing, treat as struggling." Tool-call count is a reasonable proxy for time elapsed.
- 5.3. **Token cost concern:** background sub-agents that simply wait and fire on schedule (via `ScheduleWakeup`) do NOT consume tokens while dormant; they only cost upon firing. The main concern is that frequent firings (every 3 min) accumulate turns and context — for a typical 30-min portal session that would be 10 firings, each loading the full conversation. Using ≤3 min per page (tool-call count proxy) and ≤10 min per portal (one `ScheduleWakeup` fire) balances precision against cost.

6. Issue 2.5: if `ScheduleWakeup` heartbeat is adopted for 2.3, `cc_reminder.md` retains its S0.1 purpose (re-read every card) AND gains a secondary purpose (re-read every 10 min). The file is already lightweight; no structural change is needed. The content should remain as-is; the heartbeat simply increases frequency of reads.
