## 2. Mechanical faults (NEW —— highest priority)

2. `seek/.claude/settings.json` is malformed JSON.
- 2.1. Exact defect: inside the `"hooks"` object, the sole key `"PostCompact": [ … ]` is followed by a trailing comma before the closing brace —— the file reads `],` then `},`. JSON forbids the trailing comma; `python3 -c "json.load(...)"` raises `JSONDecodeError`.
- 2.2. Consequence: when the session's project scope is `/seek/`, Claude Code cannot parse this settings file, so the `PostCompact` hook (→ `seek/.claude/post_compact.sh`) is never registered. No override text is injected at compaction. This precisely matches the `rchat` symptom "PostCompact hook either isn't configured or didn't fire".
- 2.3. Verified facts: the root `.claude/settings.json` IS valid and registers a root `PostCompact` hook; both `post_compact.sh` scripts exist and are executable (root 2054 B, seek 1030 B).

3. The two `post_compact.sh` scripts issue CONTRADICTORY post-compaction instructions.
- 3.1. Root `post_compact.sh`: "STOP. Do NOT continue any prior task. Do NOT re-read any file … Await the user's instruction" (root CLAUDE.md §5 halt-and-wait model).
- 3.2. Seek `post_compact.sh`: "re-read ALL mandatory files BEFORE any other action … DO NOT continue the loop" (AJAP re-read-then-resume model).
- 3.3. Conflict: in AJAP mode the seek model is the intended one (re-read → auto-resume). If the root hook is the one that fires (e.g. project scope = repo root), MA is told to halt and wait —— which both contradicts AJAP autonomy and the user's valued "auto-continue" behaviour. Whichever hook governs must be deterministic and aligned with AJAP.

4. No hook exists for the ordinary heartbeat path.
- 4.1. A `Monitor`/watchdog wake delivers only a plain stdout line (`heartbeat`/`newfile`/`sa2-alert`/`watchdog-alive`) surfaced as a `task-notification`. That text carries NO forcing instruction.
- 4.2. Thus the mandated "literally re-read `MA_hb.md` every wake" (`MA_hb.md` §42; `main_ajap.md` §141.1) depends entirely on MA's unaided discipline —— there is no mechanical enforcement at all on the hot path.

5. Watchdog liveness signal is structurally blind to the real failure.
- 5.1. MA touches `ma_hb_reread_marker` at the TOP of every wake, BEFORE and INDEPENDENTLY of the re-read. The SA2 watchdog only checks that marker's mtime (`find -mmin +6`).
- 5.2. Therefore a wake that touches-but-skips-the-read still shows a fresh marker → `watchdog-alive`. The watchdog proves "MA woke", never "MA re-read". It cannot detect §3.
