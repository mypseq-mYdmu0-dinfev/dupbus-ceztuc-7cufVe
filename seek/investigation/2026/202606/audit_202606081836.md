2. CRITICAL — Heartbeat wake-handler ran hollow, and it RECURRED unfixed
- 2.1. Symptom: `latest_TS` frozen at `202606081610` across ~90 subsequent ARs; user reported "heartbeat still not working" twice.
- 2.2. Behaviour: on every Monitor wake MA only `touch`ed `ma_hb_reread_marker` (and `ma_c2_marker` on C2). It did NOT Read `MA_hb.md`, run the Active Check, run the broad file check, advance `latest_TS`, nor emit `✅ context/MA_hb.md`.
- 2.3. The process was never broken: the stopped Monitors' recovered commands were exact canonical persistent forms with correct absolute paths; they fired for hours. The defect is 100% behavioural (the handler), not the infrastructure.
- 2.4. RECURRENCE is the key finding: the identical failure was logged at N=10 (rlog 202606080407) WITH a fix suggestion, then recurred and was re-logged at N=92 (rlog 202606081733). It repeated because the suggested file changes were only logged, never applied. Logging without enacting is insufficient.
- 2.5. Why the watchdog could not catch it (structural): SA2 measures only the marker's mtime. Touching the marker satisfies liveness WITHOUT the actual re-read, so `watchdog-alive` printed throughout a hollow heartbeat. The liveness proxy is decoupled from the real action.
- 2.6. Memory-feedback breaches (tied to 2.2): `feedback_hb_reread_mahb` (Read with the tool every wake) and `feedback_declare_mahb_reread` (emit `✅ context/MA_hb.md`) — both violated on every wake until the user forced a full re-read.

- 3.3. Consequence B: the `⏳_` CL-review approval gate had no independent reviewer — MA approved its own submissions implicitly; the between-loop audit was self-audit.
- 3.4. No file (`main_ajap.md`, `ajap.md`, `MA_hb.md`) specifies this hybrid, so there was no checklist to fall back on.
- 3.5. Upstream driver: SAs kept failing — repeated context-exhaustion compactions (rlog N=5/11/18/22/24), a Sonnet quota cutoff, and finally an Opus SA refusing the role. The solo mode was a symptom of unreliable SA spawning, which itself warrants investigation.
