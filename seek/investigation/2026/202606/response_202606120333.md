# Response to query_202606120333.md
*What cl_check.py is, how it differs from ajap_logs.py, and the wiring fix*

1. What `cl_check.py` does.
- 1.1. It is a per-AR PRE-FLIGHT GATE: pass it one (or a few) AR path(s); it lints just that CL and returns an exit code —— `0` clean, `1` hard violation, `2` error —— with the specific issues printed.
- 1.2. HARD checks (block a Submit): CL section heading ≠ `## 6.`; any `—` `–` `+` in the CL; `16⁺` appearing more than once; file not ending with the exact P.S. line.
- 1.3. SOFT checks (advisory, never block): colon-pillar headers (the `audit_202606040352` finding) and AI-slop/banned words —— flagged with `⚠️` because they can legitimately match a company name.
- 1.4. Purpose: give the agents a COLD, deterministic gate so a format-broken CL cannot ride through on an agent's "it looks fine" judgement.

2. How it differs from your `ajap_logs.py`.
- 2.1. Direction —— `ajap_logs.py` MEASURES retrospectively; `cl_check.py` PREVENTS in real time. One is analytics, the other is a gate.
- 2.2. Scope —— `ajap_logs.py` scans ALL of `gcl/{applied,skipped,pending}` and buckets every job into timeframes; `cl_check.py` looks at the single AR you hand it.
- 2.3. Output —— `ajap_logs.py` writes a CSV (Applied/Skipped/Pending counts + aggregate violation tallies V, VF, V/A, VF/A per timeframe); `cl_check.py` prints per-file issues and sets an exit code a Bash step can branch on.
- 2.4. Checks —— `cl_check.py` reuses your verified logic (symbols-after-CL-marker, exact P.S. tail) but adds section-heading, `16⁺`-once, colon-pillar, and the hard/soft split that `ajap_logs.py` (a counter) has no need for.
- 2.5. In short: I lifted the violation logic out of your analytics script and inverted its purpose —— from "tally what already went wrong" to "stop it before Submit".
